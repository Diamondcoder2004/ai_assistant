## 1. Root Cause: Why Binary Correctness = 25.2%

### 1.1 The Data Tells a Clear Story

Comparing binary-correct (n=70) vs binary-incorrect (n=208) answers:

| Metric | Correct (n=70) | Incorrect (n=208) | **Delta** |
|--------|---------------|-------------------|-----------|
| **Completeness** | 4.91 | 2.38 | **+2.53** |
| **Context Recall** | 4.89 | 2.77 | **+2.11** |
| Relevance | 5.00 | 4.21 | +0.79 |
| Faithfulness | 4.86 | 4.31 | +0.55 |
| Hallucination Risk | 4.86 | 4.38 | +0.48 |
| Sources Retrieved | 3.03 | 3.64 | **-0.62** |

**Conclusion:** Completeness and Context Recall are the dominant predictors. When sources contain the right information, the LLM generates correct answers. When sources are irrelevant, answers fail — regardless of generation quality (faithfulness 4.3 is fine, hallucination risk 4.4 is fine).

### 1.2 The Source Count Paradox

| Sources | Binary Correct Rate | N |
|---------|---------------------|---|
| 1 | **38%** | 42 |
| 2 | 30% | 56 |
| 3 | 27% | 49 |
| 4 | 15% | 53 |
| 5 | 23% | 35 |
| 6 | 25% | 28 |
| 7+ | **0%** | 13 |

**More retrieved chunks → LOWER correctness.** The hybrid search returns many chunks, but they're increasingly irrelevant — diluting the signal with noise. The LLM then either:
1. Gets confused by contradictory/irrelevant context (hallucination)
2. Can't extract the right information from the pile

### 1.3 The Worst 7 Cases (complete retrieval failure)

7 questions with binary=0, relevance≤2, **context_recall=1.0** (sources contain zero relevant info). These represent *total retrieval failure* — the system retrieves chunks that have NO connection to the question:

- **Q1:** "В какие сроки сетевая организация должна направить уведомление о восстановлении электроснабжения?" — expected to find regulation timelines, got unrelated content
- **Q138:** "Где в личном кабинете раздел «Дополнительные услуги»?" — expected to find LK navigation info, got nothing relevant
- **Q253:** "Где можно получить бланк заявления на перераспределение мощности?" — expected form location, retrieval failed
- **Q283:** "Какими документами регулируется проведение испытаний электроустановок?" — expected specific regulations
- **Q285/Q286:** Category clarification for ИП/ЮЛ registration — domain classification failure
- **Q299:** "Как отследить статус заявки в ЛК?" — expected LK UI steps, got unrelated operational content

## 2. Trace-Level Findings

### 2.1 WikiRouter Has NO Tracing

```
agents/search_agent.py:     @observe_rag(name="SearchAgent.search")     ✅
agents/response_agent.py:   @observe_rag(name="ResponseAgent.generate") ✅
agents/query_generator.py:  @observe_rag(name="QueryGenerator.generate") ✅
wiki/*:                     (no @observe_rag)                           ❌
```

**This is critical blind spot.** The WikiRouter is the FIRST step in the pipeline — it selects which documents to provide as context to the QueryGenerator. Without tracing, we cannot:
- See which documents WikiRouter selected for a given query
- Determine if it's filtering OUT relevant documents
- Measure WikiRouter latency per call
- Debug document selection failures

### 2.2 What Langfuse Shows for Failed Traces

From the 3080 scores attached to the benchmark root trace (`b9696ea5ffc111801d1753a6acd31c4b`):
- **SearchAgent.search** spans show what was retrieved (chunk IDs, scores)
- **QueryGenerator.generate** spans show the reformulated search queries
- **ResponseAgent.generate_response** spans show the prompt + response
- **Missing: WikiRouter.route** — we can't see document context injection

For the 7 worst cases, the SearchAgent spans would likely show:
- Retrieved chunks with irrelevant content
- Semantic search scores that don't align with BM25 scores
- Queries generated by QueryGenerator that don't match the actual information need

### 2.3 Failure Pattern: Semantic vs Lexical Mismatch

The hybrid search weights are:
```
PREF(semantic: summary+content) = 0.4
HYPE(semantic: questions)       = 0.3
LEXICAL(BM25)                   = 0.2
CONTEXTUAL(neighbor chunks)     = 0.1
```

70% of the weight is on **semantic embedding similarity**. If the embeddings of a query about "сроки восстановления электроснабжения" are closer to general FAQ content than to the specific regulation paragraph about restoration timelines, BM25 can't compensate (only 20% weight).

## 3. Systemic Issues Identified

### Issue A: Retrieval Precision < Recall
The system retrieves many chunks (avg 3.5) but they're often irrelevant. Even with 7+ chunks, binary correctness = 0%. The hybrid search is optimized for quantity, not relevance.

### Issue B: No Post-Retrieval Relevance Filter
There's no step that re-ranks or filters chunks after retrieval. The LLM gets all top-k chunks regardless of relevance score.

### Issue C: WikiRouter Unknown Impact
Without tracing, we don't know if WikiRouter helps or hurts. It might be:
- Injecting too much context (noise)
- Injecting wrong document context
- Not injecting the right documents at all

### Issue D: Embedding Model Limitations
The `perplexity/pplx-embed-v1-4b` embeddings might not capture the fine-grained distinctions between regulatory language (normative_documents) and operational language (operational_content), causing semantic drift.

### Issue E: No Domain-Specific Query Expansion
The QueryGenerator reformulates queries but doesn't add domain-specific context (e.g., client_type, power_range) that would help narrow the search.

## 4. Improvement Plan (Ordered by Impact)

### Phase 1: Add Observability (1 file, ~2h) ✅ DONE 2026-05-07

**P1.1 — Add @observe_rag to WikiRouter** ✅
- File: `backend/wiki/wiki_router.py`
- Added `@observe_rag(name="WikiRouterAgent.route")` to `route()` method
- Added `@observe_rag(name="WikiRouterAgent.llm_route")` to `_llm_route()` method
- **Impact:** Enables root-cause debugging of retrieval failures

**P1.2 — Log per-chunk relevance scores in SearchAgent traces** ✅
- File: `backend/agents/search_agent.py`
- Added `pref_score`, `hype_score`, `contextual_score`, `bm25_score` from `r.metadata` to the logging output
- **Impact:** Reveals whether semantic or lexical search is driving irrelevant results

### Phase 2: Improve Retrieval Quality (2-3 files, ~4h) — ✅ DONE 2026-05-07

**P2.1 — Implement Post-Retrieval Relevance Filter** ✅ DONE 2026-05-07
- New: `backend/tools/relevance_filter.py`
- Strategy: lemmatized token overlap (same pymorphy3 as BM25) — zero LLM latency
- Drops chunks with zero shared lemmas between query and chunk content+summary
- Wired into `search_agent.py` after `search_multi()` at line 177-183
- Default: `MIN_OVERLAP=1`, `MAX_RESULTS=5`
- **Verified working:** "Relevance filter: kept 1, dropped 8 (min_overlap=1)"

**P2.2 — Adaptive BM25 Weight Boost** ✅ DONE 2026-05-07
- File: `backend/tools/search_tool.py`
- Config: `ADAPTIVE_BM25_BOOST=true`, `ADAPTIVE_BM25_THRESHOLD=0.7` in `config.py`
- When max pref_score < 0.7, shifts 0.2 weight from HYPE → LEXICAL (0.2→0.4)
- **Verified working:** "Adaptive BM25: max pref_score=0.512 < threshold=0.7, boosting lexical weight 0.2→0.4"

**P2.3 — Add client_type + power_range to QueryGenerator context** ✅ DONE 2026-05-07
- File: `backend/main.py` (wiki_context_short construction, lines 124-135)
- Strategy: Inject WikiDocument.client_types and power_ranges into the short wiki context
- QueryGenerator LLM now sees e.g. `[ФЛ; <15kW]` per document → naturally produces domain-specific queries
- **No prompt changes needed** — the LLM picks up the metadata from context formatting
- Example output: `• Тех.присоединение [ТПП] [ФЛ; <15kW]: заявка, подключение, мощность...`

### Phase 3: Fix Specific Failure Patterns (2 files, ~3h)

**P3.1 — Add FAQ-to-Regulation Linking** ✅ DONE 2026-05-07
- When a question is about procedures/timelines, search BOTH collections with balanced weights
- Current behavior may over-prioritize operational_content (FAQ) when the answer is in normative_documents
- File: `backend/agents/search_agent.py` (step 3.6, lines ~188-201)
- Approach: `is_regulatory_query()` lemmatized keyword detection in `tools/relevance_filter.py`
- When detected: boost `normative_documents` chunks' `score_hybrid` ×1.15, re-sort
- Config: `REGULATORY_QUERY_BOOST=true`, `REGULATORY_QUERY_BOOST_FACTOR=1.15`

**P3.2 — Implement "Source Quality" Scoring in ResponseAgent** ✅ DONE 2026-05-07
- File: `backend/agents/response_agent.py` (`_create_user_prompt` method)
- Before generating response, check if retrieved chunks actually contain answer-relevant info
- Approach: `compute_source_quality()` in `tools/relevance_filter.py` blends overlap (40%) + count (20%) + hybrid_score (40%)
- If quality score < `SOURCE_QUALITY_THRESHOLD` (default 0.25), injects warning into user_prompt
- Warning: "⚠️ ВНИМАНИЕ: Качество источников для этого вопроса НИЗКОЕ... Если информации действительно недостаточно — честно укажи это, не додумывай детали."
- Config: `SOURCE_QUALITY_THRESHOLD=0.25`
- Wired through: search_agent → result dict → main.py → response_agent.generate_response()

### Phase 4: Benchmark Diagnostics (1 file, ~2h)

**P4.1 — Add Per-Source Relevance Annotations to Benchmark**
- File: `backend/benchmark.py`
- After each evaluation, log which specific chunk(s) the judge identified as containing the answer
- Store in CSV for post-hoc analysis
- **Impact:** Pinpoint exactly which retrieval steps fail
