# Отчет о проекте

## ИИ-ассистент для поддержки клиентов по вопросам технологического присоединения (Башкирэнерго)

---

## 1. Постановка задачи

Разработка интеллектуальной системы поддержки клиентов компании «Башкирэнерго» по вопросам технологического присоединения. Система должна автоматически отвечать на вопросы клиентов, используя базу знаний нормативных документов, и предоставлять ответы в режиме реального времени через веб-интерфейс.

**Ключевые требования:**
- Автоматизация ответов на типовые вопросы по технологическому присоединению
- Снижение нагрузки на операторов колл-центра
- Круглосуточная доступность
- Фиксация истории обращений и обратной связи для аналитики

---

## 2. Выбор архитектуры и подхода к решению

### 2.1. Почему RAG, а не дообучение модели

В качестве основного подхода была выбрана архитектура **RAG (Retrieval-Augmented Generation)** — генерация с извлечением из базы знаний. Причины:

- **Отсутствие вычислительных ресурсов** для дообучения (fine-tuning) языковой модели на корпоративных данных
- **Актуальность знаний:** нормативная база обновляется, и RAG позволяет обновлять ответы без переобучения модели — достаточно обновить базу документов
- **Объяснимость:** каждый ответ подкреплён конкретными источниками из базы знаний, что критично для юридически значимой области
- **Контроль галлюцинаций:** модель ограничена предоставленными документами, что снижает риск выдуманных ответов

### 2.2. Обзор рынка и анализ подходов

При выборе архитектуры был проведён анализ существующих решений:

- **Готовые RAG-чатботы** (LangChain, LlamaIndex) — предоставляют базовый функционал, но не учитывают специфику домена
- **Agentic RAG** — использование нескольких специализированных агентов для переформулирования запроса, поиска и генерации ответа. Этот подход показал лучшие результаты по качеству поиска
- **Hybrid Search** — комбинация векторного и лексического поиска (BM25) для повышения релевантности
- **Без реранкера** — отказ от отдельного этапа реранкинга в пользу гибридного скоринга с весами, что упрощает пайплайн и снижает задержку

## 3. Выбор языковых моделей

### 3.1. Критерии выбора

Выбор моделей основывался на трёх ключевых факторах:
1. **Скорость работы** — время ответа должно быть приемлемым для пользователя
2. **Стоимость** — использование платного сервиса RouterAI
3. **Качество генерации и эмбеддингов** — точность ответов и релевантность поиска

### 3.2. Выбранные модели

| Компонент | Модель | Назначение |
|-----------|--------|------------|
| **Генератор (LLM)** | `inception/mercury-2` | Генерация ответов, переформулирование запросов, генерация гипотетических вопросов |
| **Эмбеддинг-модель** | `pplx-embed-v1-4b` | Векторное кодирование чанков и запросов |

**Mercury 2** — диффузионная языковая модель, обеспечивающая высокую скорость генерации. Несмотря на то, что в пайплайне LLM вызывается 3–4 раза на один запрос, общее время ответа остаётся приемлемым благодаря диффузионной архитектуре.

**Метод выбора эмбеддинг-модели** основывался на тестировании через **top-k accuracy** с использованием скрипта автоматизированного тестирования. Дополнительно применялась генерация гипотетических вопросов для каждого чанка, что повысило релевантность семантического поиска.

## 4. Архитектура системы

### 4.1. Компоненты системы

Система состоит из следующих модулей:

1. **Веб-интерфейс** (Vue.js 3) — интерфейс для взаимодействия с клиентом
2. **Модуль авторизации и регистрации** (Supabase Auth) — JWT-аутентификация пользователей
3. **Модуль ИИ и поиска** (FastAPI + агенты) — обработка запросов, поиск, генерация ответов
4. **База знаний** (Qdrant) — векторное хранилище для семантического поиска
5. **База данных пользователей и истории чатов** (Supabase PostgreSQL) — хранение сессий, вопросов, ответов, обратной связи

### 4.2. Зависимость от внешнего сервиса

Система работает с платным сервисом **RouterAI**, который предоставляет доступ к:
- Языковой модели для генерации ответов (mercury-2)
- Эмбеддинг-модели для векторного поиска

**Важно:** запросы пользователей передаются в облако RouterAI, что создаёт риски для конфиденциальности персональных данных. Этот вопрос требует отдельного рассмотрения.

## 5. Формирование базы знаний

### 5.1. Стратегия фрагментации (Chunking)

Документы нормативной базы обрабатываются в несколько этапов:

1. **Разбиение по заголовкам** с рекурсивным дроблением крупных секций
2. **LLM-обогащение** каждого чанка:
   - Генерация краткого содержания (summary)
   - Генерация гипотетических вопросов, на которые чанк может ответить
   - Извлечение ключевых слов и сущностей
3. **Валидация и восстановление JSON** при ошибках парсинга

### 5.2. Стратегия кодирования (3 вектора на чанк)

Каждый чанк кодируется **тремя векторами** в Qdrant:

| Вектор | Содержание | Назначение |
|--------|------------|------------|
| **pref** | summary + content | Семантический поиск по краткому содержанию и тексту |
| **hype** | hypothetical questions | Поиск по гипотетическим вопросам |
| **contextual** | prev + current + next чанк | Учёт контекста соседних фрагментов |

---

## 6. Детали поиска и генерации: агенты

### 6.1. Мультиагентная архитектура

Система использует **три агента**, работающих последовательно:

1. **Query Generator** — переформулирует запрос пользователя с учётом предыдущих сообщений в сессии, разбивает сложные вопросы на составные части
2. **Search Agent** — выполняет гибридный поиск с настраиваемыми весами в зависимости от сложности запроса
3. **Response Agent** — генерирует финальный ответ на основе найденных источников

### 6.2. Гибридный поиск

Поиск комбинирует **4 компонента** с настраиваемыми весами:

| Компонент | Вес | Описание |
|-----------|-----|----------|
| **Семантический (pref)** | 0.4 | Поиск по вектору summary+content |
| **Семантический (hype)** | 0.3 | Поиск по вектору гипотетических вопросов |
| **Семантический (contextual)** | 0.1 | Поиск по вектору соседних чанков |
| **Лексический (BM25)** | 0.2 | Поиск по словесному совпадению |

**Формула:** `hybrid = 0.4×pref + 0.3×hype + 0.1×contextual + 0.2×bm25`

Веса контролируются агентом и могут варьироваться в зависимости от сложности и типа запроса.

### 6.3. Система уточнения вопросов

Если вопрос пользователя не относится к технологическому присоединению или является некорректным, система запрашивает уточнение. При отсутствии релевантных источников в базе знаний предусмотрен **fallback к оператору** (после одного повторного вопроса).

## 7. Функциональные возможности

- **История сообщений** — сохранение всех вопросов и ответов в базу данных для аналитики
- **Возобновление предыдущих диалогов** — пользователь может вернуться к любой сессии
- **Обратная связь** — пользователи могут оценить качество ответа
- **Источники ответа** — каждый ответ содержит ссылки на документы-источники
- **Горячие клавиши** — ускоренная работа с интерфейсом (Ctrl+N, Ctrl+H, Ctrl+L и др.)

## 8. Преимущества и недостатки

### 8.1. Недостатки (подлежащие исправлению)

| Недостаток | Описание |
|------------|----------|
| **Отсутствие кэширования запросов** | На один и тот же вопрос модель отвечает заново, расходуя токены и время |
| **Риск некорректных ответов** | Если модель ответила неправильно, ответ сохраняется и показывается пользователям |
| **Необходимость работы с фидбеком** | Система сбора обратной связи реализована, но механизмы автоматического улучшения ещё не внедрены |
| **Зависимость от внешнего сервиса** | Данные запросов уходят в облако RouterAI — проблема безопасности и конфиденциальности |
| **Отсутствие автоматического обновления базы знаний** | При обновлении нормативной базы документы нужно обновлять вручную |
| **Обновление может выполнить только программист** | Нет интерфейса для самостоятельного обновления базы знаний |

### 8.2. Преимущества

| Преимущество | Описание |
|--------------|----------|
| **Низкие требования к ресурсам** | Не требуется GPU для дообучения, система работает на CPU |
| **ИИ-агенты** | Переформулирование запросов, разбиение сложных вопросов, адаптация параметров поиска |
| **Гибридный поиск по 3 векторам + BM25** | Высокая релевантность результатов |
| **Быстрое время ответа** | Благодаря диффузионной модели mercury-2 |
| **Система аналитики** | Запись всех чатов и пользователей для последующего анализа |
| **Обратная связь с клиентами** | Возможность оценки качества ответов |
| **Система уточнения вопросов** | Фильтрация нерелевантных запросов |

## 9. Технические требования и инфраструктура

### 9.1. Минимальные требования к серверу

| Ресурс | Требование |
|--------|------------|
| **Процессор** | 4 ядра |
| **Оперативная память** | 4–8 ГБ |
| **Дисковое пространство** | от 10 ГБ (зависит от объёма логов и базы чатов) |

### 9.2. Финансовые затраты

Основные расходы — подписка на сервис **RouterAI** для доступа к языковым моделям.

### 9.3. Контейнеризация

Система развёрнута через **Docker Compose**:
- Backend: FastAPI + Uvicorn
- Frontend: Vue.js 3 + Nginx
- Qdrant: векторная база данных
- Supabase: PostgreSQL + Auth

---

## 10. Условия тиражирования

Для развёртывания системы на внутреннем контуре заказчика необходимо:

1. Предоставить сервер с указанными выше характеристиками
2. Настроить Docker и Docker Compose
3. Развернуть контейнеры из репозитория
4. Настроить подключение к внутреннему Supabase (при необходимости)
5. При желании — заменить RouterAI на локальную языковую модель (требует GPU)

**Документация для развёртывания** включена в репозиторий: `README.md`, `QWEN.md`, `docker-compose.yml`, `.env.example`.

---

## 11. Заключение

Разработанная система ИИ-ассистента для поддержки клиентов по вопросам технологического присоединения демонстрирует работоспособность подхода **Agentic RAG** с гибридным поиском. Система готова к тестированию и внедрению, при этом существует ряд направлений для улучшения: кэширование запросов, автоматическое обновление базы знаний, локализация языковых моделей для повышения безопасности данных.
