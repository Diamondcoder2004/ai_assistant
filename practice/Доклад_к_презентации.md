# Доклад к презентации: ИИ-ассистент по технологическому присоединению

---

## Слайд 1: Титульный слайд

**ИИ-ассистент по технологическому присоединению**

Представлен проект интеллектуального чат-бота для автоматизации поддержки клиентов компании Башкирэнерго по вопросам технологического присоединения. Система построена на базе RAG-архитектуры с использованием современных технологий искусственного интеллекта.

---

## Слайд 2: Актуальность работы

Энергетическая отрасль сталкивается с растущим потоком обращений клиентов по вопросам технологического присоединения. Традиционные методы поддержки не справляются с объемом запросов, что приводит к увеличению времени ожидания и снижению качества обслуживания.

Ежемесячно обрабатывается более 1000 обращений, при этом около 40% из них составляют повторяющиеся вопросы. Операторы тратят значительное время на поиск информации в нормативных документах, что снижает общую эффективность работы.

Цифровизация энергетического сектора требует внедрения автоматизированных систем поддержки, способных работать в режиме 24/7 и обеспечивать мгновенные консультации.

---

## Слайд 3: Проблема заказчика

Выявлен ряд критических проблем в текущем процессе обслуживания:

Высокая нагрузка на операторов колл-центра приводит к профессиональному выгоранию и снижению качества консультаций. Длительное время ожидания ответа клиента снижает удовлетворенность сервисом.

Отсутствует единая централизованная база знаний по технологическому присоединению. Навигация по нормативно-правовым актам представляет сложность как для клиентов, так и для операторов.

Необходимость быстрого поиска актуальной информации в нормативных документах требует значительных временных затрат.

---

## Слайд 4: Цель работы

Определена основная цель — разработка интеллектуального чат-бота для автоматизации консультаций по технологическому присоединению.

Для достижения цели реализованы следующие задачи:

Внедрен гибридный поиск, комбинирующий семантический и лексический подходы для повышения точности результатов. Выполнена интеграция с векторной базой данных Qdrant для хранения и поиска эмбеддингов документов.

Обеспечена надежная аутентификация пользователей через платформу Supabase с поддержкой JWT-токенов. Создан интуитивный веб-интерфейс на базе Vue.js 3 с поддержкой streaming-отображения ответов.

Реализована streaming-генерация ответов через LLM с использованием серверных отправленных событий (SSE).

## Слайд 5: Обзор аналогов продукта

Проведен анализ существующих решений на рынке. Универсальные чат-боты не поддерживают RAG-архитектуру и гибридный поиск, а также не обладают отраслевой специализацией.

Базы знаний с поиском предоставляют только лексический поиск без семантического понимания запросов и не используют генеративные возможности LLM.

Разработанная система сочетает RAG-архитектуру с 4-компонентным гибридным поиском, специально адаптированным для энергетической отрасли с учетом нормативной базы Башкирэнерго.

## Слайд 6: Функциональная модель (Use Case)

Определены основные сценарии взаимодействия клиента с системой. Клиент отправляет запрос по вопросам технологического присоединения через веб-интерфейс.

Система обрабатывает запрос через RAG-архитектуру с использованием LLM и векторной базы данных Qdrant. Клиент получает развернутый ответ с указанием источников информации.

Реализована возможность просмотра истории чатов для отслеживания предыдущих обращений. Внедрена система оценки качества ответов через механизмы like/dislike.

## Слайд 7: Логическая модель базы данных

Спроектирована реляционная схема базы данных на базе PostgreSQL через платформу Supabase.

Таблица users хранит данные аутентификации, включая идентификатор, email, зашифрованный пароль и метаданные пользователей.

Таблица chat_sessions содержит информацию о сессиях общения с привязкой к пользователям, заголовками и временными метками.

Таблица messages хранит сообщения с указанием роли (пользователь или ассистент), содержимым, источниками в формате JSONB и временными метками.

Таблица feedback фиксирует оценки пользователей для последующего анализа качества системы.

Все таблицы связаны через внешние ключи, обеспечивая целостность данных.

## Слайд 8: UML диаграмма классов

Спроектирована объектная модель системы, включающая шесть основных классов.

Класс SearchAgent отвечает за генерацию поисковых запросов, поиск документов и ранжирование результатов. Класс ResponseAgent выполняет генерацию ответа, извлечение источников и форматирование вывода.

Класс SearchTool реализует гибридный поиск с нормализацией BM25 и расчетом весовых коэффициентов. Класс ChatService обрабатывает запросы, организует streaming ответов и сохранение сессий.

Класс AuthService управляет аутентификацией, регистрацией и валидацией токенов. Класс EmbeddingService выполняет векторизацию текста и взаимодействие с Qdrant.

## Слайд 9: Жизненный цикл ПО

Применена итеративная модель разработки с непрерывным циклом обратной связи.

Жизненный цикл включает шесть фаз: анализ требований, проектирование, разработка, тестирование, деплой и поддержка. Каждый этап включает обязательное тестирование и валидацию результатов.

Реализованы практики CI/CD через Docker Compose для автоматизации развертывания. Внедрено автоматическое тестирование API для обеспечения стабильности работы.

Организован мониторинг через систему логирования всех компонентов. Выполняются регулярные обновления модели для повышения точности ответов.

## Слайд 10: Модель векторной базы данных

Развернута векторная база данных Qdrant с коллекцией BASHKIR_ENERGO_PERPLEXITY.

Используется модель эмбеддингов perplexity/pplx-embed-v1-4b с размерностью вектора 2560. Для измерения схожести применяется Cosine similarity.

Индекс HNSW (Hierarchical Navigable Small World) обеспечивает эффективный поиск в многомерном пространстве.

Структура payload каждого вектора включает: текст документа, краткое содержание, гипотетические вопросы, источник документа и дополнительные метаданные.

Гибридный поиск объединяет четыре компонента: семантический поиск по содержанию (40%), семантический поиск по гипотетическим вопросам (30%), лексический поиск BM25 (20%) и контекстный поиск соседних чанков (10%).

## Слайд 11: Архитектура системы

Спроектирована трехзвенная архитектура с разделением ответственности.

Браузер клиента взаимодействует с системой через веб-интерфейс на Vue.js 3. Запросы проходят через Nginx, выполняющий функции reverse proxy на порту 80.

Backend реализован на базе FastAPI и работает на порту 8880. Обеспечивает обработку запросов, управление сессиями и интеграцию с внешними сервисами.

Supabase предоставляет реляционную базу данных PostgreSQL и сервис аутентификации. Qdrant выполняет векторный поиск на порту 6333. RouterAI API обеспечивает доступ к LLM и сервису эмбеддингов.

## Слайд 12: Компоненты системы

Система состоит из шести ключевых компонентов.

Search Agent генерирует поисковые запросы через LLM, выполняет гибридный поиск и ранжирует результаты по релевантности. Response Agent получает результаты поиска, генерирует ответ через LLM, извлекает источники и форматирует вывод.

Search Tool реализует 4-компонентный поиск с нормализацией BM25 и взвешиванием результатов. Chat Store управляет состоянием чата, хранит историю сессий и организует streaming данных.

Auth Service обеспечивает JWT-аутентификацию, регистрацию и вход пользователей через Supabase. Embedding Service выполняет векторизацию текста и подключение к Qdrant.

---

## Слайд 13: Пайплайн обработки данных

Процесс обработки запроса включает шесть последовательных этапов.

На первом этапе клиент вводит вопрос в чат-бот. На втором этапе LLM генерирует варианты поисковых запросов для расширения охвата.

На третьем этапе выполняется гибридный поиск в Qdrant по четырем компонентам. На четвертом этапе происходит нормализация результатов и взвешивание по формуле.

На пятом этапе LLM генерирует ответ на основе найденного контекста документов. На шестом этапе ответ передается клиенту через потоковую передачу SSE.

Формула гибридного поиска: hybrid = 0.4 × pref + 0.3 × hype + 0.1 × contextual + 0.2 × BM25. Нормализация BM25 выполняется по классической формуле: score / max_score.

## Слайд 14: Инструменты разработки

Для реализации проекта применен современный стек технологий.

Backend разработан на Python 3.11 с использованием фреймворка FastAPI. Сервер Uvicorn обеспечивает асинхронную обработку запросов. Pydantic выполняет валидацию данных.

Frontend создан на Vue.js 3 со сборщиком Vite. Pinia управляет состоянием приложения. Vue Router обеспечивает навигацию между страницами.

Базы данных: Supabase для аутентификации и PostgreSQL, Qdrant для векторного поиска, pgvector для работы с эмбеддингами.

Инфраструктура: Docker и Docker Compose для контейнеризации, Nginx как reverse proxy, RouterAI API для доступа к моделям.
