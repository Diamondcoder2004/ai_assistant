# ИИ-ассистент для поддержки клиентов по вопросам технологического присоединения
# БашкирЭнерго

# Содержание

# 1. Постановка задачи
Разработка интеллектуальной системы поддержки клиентов компании «БашкирЭнерго» по вопросам технологического присоединения. Система должна автоматически отвечать на вопросы клиентов, используя базу знаний нормативных документов, и предоставлять ответы в режиме реального времени через веб-интерфейс.
Ключевые требования:
Автоматизация ответов на типовые вопросы по технологическому присоединению
Снижение нагрузки на операторов колл-центра
Круглосуточная доступность
Фиксация истории обращений и обратной связи для аналитики
# 2. Выбор архитектуры и подхода к решению
## 2.1. Почему RAG, а не дообучение модели
В качестве основного подхода была выбрана архитектура RAG (Retrieval-Augmented Generation) — генерация с извлечением из базы знаний. Причины:
Отсутствие вычислительных ресурсов для дообучения (fine-tuning) языковой модели
Актуальность знаний: нормативная база обновляется, и RAG позволяет обновлять ответы без переобучения модели
Объяснимость: каждый ответ подкреплён конкретными источниками
Контроль галлюцинаций: модель ограничена предоставленными документами
## 2.2. Обзор рынка и анализ подходов
При выборе архитектуры был проведён анализ существующих решений:
Готовые RAG-чатботы (LangChain, LlamaIndex) — базовый функционал, но без учёта специфики домена
Agentic RAG — несколько специализированных агентов для переформулирования запроса, поиска и генерации ответа
Hybrid Search — комбинация векторного и лексического поиска (BM25)
Без реранкера — отказ от отдельного этапа реранкинга в пользу гибридного скоринга с весами
# 3. Выбор языковых моделей
## 3.1. Критерии выбора
Скорость работы — время ответа должно быть приемлемым для пользователя
Стоимость — использование платного сервиса RouterAI
Качество генерации и эмбеддингов
## 3.2. Выбранные модели
Mercury 2 — диффузионная языковая модель, обеспечивающая высокую скорость генерации. Несмотря на 3–4 вызова LLM на один запрос, общее время ответа остаётся приемлемым.
Метод выбора эмбеддинг-модели основывался на тестировании через top-k accuracy с использованием скрипта автоматизированного тестирования.
# 4. Архитектура системы
## 4.1. Компоненты системы
Веб-интерфейс (Vue.js 3) — интерфейс для взаимодействия с клиентом
Модуль авторизации и регистрации (Supabase Auth) — JWT-аутентификация
Модуль ИИ и поиска (FastAPI + агенты) — обработка запросов, поиск, генерация
База знаний (Qdrant) — векторное хранилище для семантического поиска
База данных пользователей и истории чатов (Supabase PostgreSQL)
## 4.2. Зависимость от внешнего сервиса
Система работает с платным сервисом RouterAI, который предоставляет доступ к языковым моделям. Запросы пользователей передаются в облако RouterAI, что создаёт риски для конфиденциальности персональных данных.

Валидация и восстановление JSON при ошибках парсинга

# 6. Детали поиска и генерации: агенты
## 6.1. Мультиагентная архитектура
Query Generator — переформулирует запрос с учётом предыдущих сообщений
Search Agent — выполняет гибридный поиск с настраиваемыми весами
Response Agent — генерирует финальный ответ на основе источников
## 6.2. Гибридный поиск

Формула: hybrid = 0,4×pref + 0,3×hype + 0,1×contextual + 0,2×bm25
## 6.3. Система уточнения вопросов
Если вопрос не относится к технологическому присоединению, система запрашивает уточнение. При отсутствии релевантных источников предусмотрен fallback к оператору.
# 7. Функциональные возможности
История сообщений — сохранение всех вопросов и ответов
Возобновление предыдущих диалогов
Обратная связь — оценка качества ответа
Источники ответа — ссылки на документы-источники
Горячие клавиши — ускоренная работа с интерфейсом

# 11. Заключение
Разработанная система ИИ-ассистента демонстрирует работоспособность подхода Agentic RAG с гибридным поиском. Система готова к тестированию и внедрению, при этом существует ряд направлений для улучшения: кэширование запросов, автоматическое обновление базы знаний, локализация языковых моделей для повышения безопасности данных.
